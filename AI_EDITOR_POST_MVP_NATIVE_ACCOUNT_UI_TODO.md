# AI Editor MVP 后原生账号管理界面待办

更新时间：2026-07-15
状态：**已确认延期，MVP 完成后评估实施**

## 1. 当前产品决策

AI Editor 产品账号与管理页面的 MVP 采用以下方案：

- Gateway 提供统一的账号与管理 Web UI；
- 管理 Web UI 使用 React + TypeScript + Vite 构建为 Gateway 静态资源，同时保留
  `codex_proxy` 现有 standalone 管理页；
- Code 使用专用的“AI Editor 管理”Webview 标签页内嵌该页面；
- 标签页固定到受信任的 Gateway 管理地址，不显示通用地址栏、前进、后退等浏览器控件；
- 页面入口位于 AI Editor 左下角的用户头像/用户信息菜单，不位于服务器状态栏；
- 普通用户、二级管理员和一级管理员复用同一页面，由 Gateway API 强制执行角色和组织权限；
- 登录、退出和账号摘要仍使用 Code 原生入口；
- Code 使用当前设备会话申请一次性短期票据，Webview 通过 POST 换取 HttpOnly 管理
  会话；票据和产品 Token 不进入 URL 或 localStorage，Refresh Token 不进入 Webview。
- 关闭专用管理标签页只结束该 Webview 的短期管理会话，不退出产品账号；重新打开时
  自动申请新票据。
- 专用管理 Webview 只允许配置的 Gateway 管理源；外部链接交给系统默认浏览器，并
  阻止任意跨源跳转、新窗口和未经允许的下载。

该方案用于优先跑通 Windows/macOS 共用的产品账号、组织、积分、审计和中央网关 MVP。

## 2. MVP 后待评估方案

MVP 验证完成后，评估把普通用户高频账号功能改造为 Code 原生 Workbench 界面，包括：

- 我的账号资料、账号状态和有效期；
- 当前可用积分与本月使用摘要；
- 设备会话查看与撤销；
- 修改密码、退出登录和账号服务重试；
- 原生主题、键盘导航、无障碍和中文本地化。

复杂管理能力可以继续保留在 Gateway Web UI：

- 组织、用户和邀请码管理；
- 月度积分池和用户积分分配；
- 调用审计与使用统计；
- Provider、模型路由、熔断、凭据和系统诊断。

## 3. 实施前提

原生界面改造不阻塞当前 MVP。只有满足以下条件后才进入实施：

1. MVP 的登录、令牌轮换、角色权限和管理 API 已稳定；
2. 已收集普通用户对内嵌管理页的真实使用反馈；
3. 已明确哪些高频功能值得承担 Code 与 Gateway 两套界面的维护成本；
4. 原生界面仍复用同一 Gateway API，不复制权限或计费业务逻辑；
5. 开发版和 Windows/macOS 成品可以同步完成回归验证。

## 4. 验收方向

- 普通用户无需打开 Webview 即可完成高频账号操作；
- 原生界面与 Code 主题、缩放、键盘操作和无障碍能力一致；
- Gateway API 继续作为权限和数据事实来源；
- Webview 与原生界面不会产生账号状态、积分或设备会话不一致；
- 二级管理员和一级管理员的敏感权限仍由服务端强制隔离。
