# Feature Specification: 双模式 AI 编辑器

**Feature Branch**: `001-dual-mode-editor`

**Created**: 2026-06-28

**Status**: Draft

**Input**: User description: "帮我根据 VSCode 开源项目，做一个类似 Cursor 的软件，会有一个自带的 AI 对话框，跑的是 Codex 插件，会分两种模式一种是传统的 VSCode 面向研发人员的页面结构，另一种是面向业务人员的简约模式，简约模式下只显示文件目录下的文件和一个对话页面（也就是和开发模式一样的 AI 对话框），在简约模式下 AI 对话框是常驻的，可以给用户打开文件阅览文件。需要有登录接口给用户登录。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 用户登录与身份认证 (Priority: P1)

用户启动应用后，首先看到登录界面。用户输入凭据完成认证，系统验证通过后进入主界面。未登录用户无法使用任何功能。

**Why this priority**: 登录是所有功能的前置条件，没有认证就无法区分用户、保存偏好、管理会话。

**Independent Test**: 可通过启动应用 → 输入凭据 → 进入主界面的完整流程独立验证，无需其他功能模块。

**Acceptance Scenarios**:

1. **Given** 用户首次启动应用，**When** 看到登录页面，**Then** 显示凭据输入框和登录按钮，以及"使用邀请码注册"入口
2. **Given** 用户持有有效邀请码，**When** 点击注册并输入邀请码、用户名、密码，**Then** 账户创建成功并自动登录进入主界面
3. **Given** 用户输入无效或已使用的邀请码，**When** 提交注册，**Then** 显示"邀请码无效或已被使用"错误提示
4. **Given** 用户输入有效凭据，**When** 点击登录，**Then** 进入主界面（默认开发模式）
5. **Given** 用户输入无效凭据，**When** 点击登录，**Then** 显示错误提示，保持在登录页
6. **Given** 用户已登录，**When** 关闭并重新打开应用，**Then** 自动恢复登录状态（会话保持）
7. **Given** 用户会话令牌即将过期，**When** 用户仍在活跃使用中，**Then** 系统静默刷新令牌，不弹出登录页打断操作

---

### User Story 2 - 开发模式（传统 IDE 视图）(Priority: P1)

研发人员登录后使用开发模式，获得完整的 IDE 功能：文件浏览器、代码编辑器（含语法高亮和补全）、终端面板、搜索面板、以及可切换显示的 AI 对话框。

**Why this priority**: 开发模式是面向核心用户（开发者）的主要工作界面，与简约模式共享底层编辑器能力。

**Independent Test**: 登录后切换到开发模式 → 打开文件 → 编辑代码 → 打开终端执行命令 → 调出 AI 对话框提问，完整流程可独立验证。

**Acceptance Scenarios**:

1. **Given** 用户在开发模式下，**When** 浏览文件树并点击文件，**Then** 在主编辑区打开该文件并提供语法高亮
2. **Given** 用户在开发模式下，**When** 点击 AI 对话图标或使用快捷键，**Then** 侧边或底部弹出 AI 对话框
3. **Given** AI 对话框已打开，**When** 用户点击编辑器区域，**Then** AI 对话框不会自动消失（可手动关闭）
4. **Given** 用户在开发模式下，**When** 打开终端面板，**Then** 在编辑器下方显示命令行终端

---

### User Story 3 - 简约模式（业务人员视图）(Priority: P1)

业务人员登录后使用简约模式，界面只包含左侧文件目录树和右侧常驻 AI 对话框。用户可通过文件树浏览项目文件，点击文件后在 AI 对话框中查看文件内容，或直接在 AI 对话框中用自然语言描述需求、让 AI 帮助理解和操作文件。

**Why this priority**: 简约模式是本产品区别于 VSCode 的核心差异化功能，让非技术用户也能通过自然语言与代码项目交互。

**Independent Test**: 登录后切换到简约模式 → 浏览文件树 → 点击文件在对话框中查看 → 用自然语言提问，完整流程可独立验证。

**Acceptance Scenarios**:

1. **Given** 用户在简约模式下，**When** 界面加载完成，**Then** 仅显示左侧文件目录树和右侧 AI 对话框（无编辑器、终端、菜单栏等开发工具）
2. **Given** 用户在简约模式下，**When** 点击文件树中的文件，**Then** 简易编辑器显示文件内容，用户可进行基础修改和保存
3. **Given** 用户在简约模式下，**When** 在 AI 对话框中输入"请帮我看看这个项目的入口文件是什么"，**Then** AI 分析项目结构并给出回答
4. **Given** 用户在简约模式下，**When** AI 对话进行中，**Then** 对话框始终占据右侧区域，不被文件树遮挡

---

### User Story 4 - 模式切换 (Priority: P2)

用户可以通过菜单随时在开发模式和简约模式之间自由切换，无需管理员授权。切换不会中断当前 AI 对话，也不会关闭已打开的文件。首次登录默认进入用户偏好中设置的模式。

**Why this priority**: 模式切换是连接两类用户和场景的桥梁，但可在核心模式功能就绪后实现。

**Independent Test**: 在任一模式下 → 打开文件或进行 AI 对话 → 切换模式 → 验证 AI 对话保持且状态一致。

**Acceptance Scenarios**:

1. **Given** 用户在开发模式下已打开 AI 对话并进行了问答，**When** 切换到简约模式，**Then** 对话历史保持完整，简约模式继续显示该对话
2. **Given** 用户在简约模式下查看某个文件，**When** 切换到开发模式，**Then** 该文件在编辑器中打开以供编辑
3. **Given** 用户在任一模式下，**When** 通过菜单或快捷键切换模式，**Then** 切换在 1 秒内完成，无明显闪烁

---

### User Story 5 - AI 对话框交互 (Priority: P2)

AI 对话框支持多轮对话、文件引用、Markdown 渲染和代码高亮显示。用户可引用当前打开的文件作为上下文发送给 AI。AI 可以建议并帮用户打开特定文件。

**Why this priority**: AI 对话框是两种模式的核心共用组件，基础问答在 P1 已覆盖，增强交互放在 P2。

**Independent Test**: 打开 AI 对话框 → 发送消息 → 收到格式化回复 → 引用文件让 AI 分析 → 验证 AI 可操作文件。

**Acceptance Scenarios**:

1. **Given** 用户在 AI 对话框中，**When** 发送一条消息，**Then** AI 回复以 Markdown 格式渲染，代码块有语法高亮
2. **Given** 用户在 AI 对话框中，**When** 引用一个文件路径，**Then** AI 能读取该文件内容并结合上下文回答
3. **Given** AI 回复中建议打开某文件，**When** 用户点击文件链接，**Then** 在对应模式中打开该文件（开发模式进编辑器，简约模式进预览）
4. **Given** 网络断开或 AI 服务不可用，**When** 用户发送消息，**Then** 显示清晰的错误提示，不丢失已输入内容

---

### User Story 6 - 管理后台（管理员视图）(Priority: P3)

管理员通过 Web 管理后台查看系统运行数据，包括用户注册与活跃统计、AI 用量计量报表、版本推送管理、Skill 市场内容审核与发布。管理后台独立于编辑器客户端运行。

**Why this priority**: 管理后台是运营支撑功能，不影响核心用户流程，可在主功能完成后迭代。

**Independent Test**: 管理员登录后台 → 查看用户统计数据 → 推送新版本 → 审核 Skill 提交，无需依赖编辑器客户端即可验证。

**Acceptance Scenarios**:

1. **Given** 管理员使用与普通用户相同的登录入口，**When** 凭管理员账户凭据登录后，**Then** 进入管理后台仪表盘，显示用户注册数、日活跃用户数、AI 调用量等关键指标
2. **Given** 管理员在后台，**When** 生成邀请码（单个或批量），**Then** 系统生成唯一码值，可选择设置有效期和使用次数限制
3. **Given** 管理员上传新版本包，**When** 填写版本号和更新日志并推送，**Then** 所有客户端收到更新通知
4. **Given** 开发者提交 Skill 到市场，**When** 管理员审核通过，**Then** 该 Skill 在市场中对所有用户可见

---

### Edge Cases

- 用户在简约模式下没有可浏览的文件（空项目/空目录）时，AI 对话框显示引导提示
- 登录令牌过期后，用户在操作中收到重新登录提示，正在进行的 AI 对话不丢失；会话过期采用静默续期策略，只在令牌无法刷新时才要求重新登录，避免频繁打断用户操作
- 两个模式同时存在时，切换模式不丢失各自的 UI 状态（如文件树的展开状态）
- AI 返回的内容超过一屏时，对话框正确滚动，不出现渲染性能问题
- 文件名或路径包含空格、中文、特殊字符时，两种模式都能正确识别和打开

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 提供用户登录界面，支持凭据验证和会话管理
- **FR-002**: System MUST 支持用户名+密码账户登录，包含注册、登录、会话管理功能。注册采用邀请制——用户须凭管理员生成的邀请码完成注册，不支持开放自助注册。认证由独立后台服务器处理。v2 按需迭代 OAuth2 第三方登录
- **FR-003**: System MUST 提供开发模式，包含文件浏览器、代码编辑器、终端面板、搜索面板
- **FR-004**: System MUST 提供简约模式，仅包含文件目录树和常驻 AI 对话框
- **FR-005**: System MUST 在简约模式下支持点击文件后显示文件内容，并提供基础编辑能力（修改文本、保存文件）
- **FR-006**: System MUST 提供模式切换功能，切换时保持 AI 对话连续性
- **FR-007**: System MUST 集成 Codex 插件满血能力，包含多轮对话、代码生成与补全、代码审查、错误诊断与修复建议、文件读写操作、终端命令执行
- **FR-008**: AI 对话框 MUST 能够读取用户指定的文件内容作为对话上下文
- **FR-009**: AI 对话框 MUST 支持在回复中引用文件，用户点击后可以打开对应文件
- **FR-010**: System MUST 在简约模式下提供简易编辑器，支持基础文本编辑（修改、保存），无需语法高亮和代码补全等高级编辑功能
- **FR-011**: System MUST 在开发模式下支持代码语法高亮、自动补全和基础编辑功能
- **FR-012**: System MUST 保持登录会话，应用重启后自动恢复；会话过期时采用静默续期策略（如刷新令牌），只在续期失败时才要求用户重新登录，不得在单次使用周期内频繁弹出登录页打断用户
- **FR-013**: System MUST 在 AI 服务不可用时显示可操作的错误信息
- **FR-014**: System MUST 部署独立后台服务器，负责：用户认证与登录会话管理、AI 用量计量（调用次数、模型类型、token 消耗、时间戳 + 每条对话的前 50 字摘要）、版本更新推送、Skill 工具市场发布与分发
- **FR-015**: 后台服务器 MUST 仅存储登录凭证、邀请码、用量计量数据（调用统计 + 对话摘要），不存储完整对话内容、用户项目文件内容或代码数据
- **FR-016**: 后台服务器 MUST 提供管理员账户，管理员由后台直接创建（不经邀请码流程），使用与普通用户相同的登录入口，通过角色标记区分权限。管理员可查看系统用量数据、用户活跃统计、版本推送管理
- **FR-017**: 客户端 AI 对话能力 MUST 适配当前本地 Codex 插件/本地能力框架的协议、命令桥接和认证方式
- **FR-018**: System MUST 支持后台服务器向客户端推送新版本通知，客户端可触发下载更新

### Key Entities

- **用户账户 (User Account)**: 用户身份信息，包含登录凭据、偏好设置（默认模式、会话历史）
- **邀请码 (Invitation Code)**: 管理员生成的注册凭证，包含唯一码值、生成时间、状态（未使用/已使用）、使用者账户
- **项目会话 (Project Session)**: 用户打开的项目/工作区，关联文件树结构和当前模式
- **AI 对话 (AI Conversation)**: 用户与 AI 的多轮对话记录。完整对话历史优先由 Codex 插件本地存储机制管理，客户端通过 Codex 接口或命令桥接读写。后台仅存储摘要元数据（前 50 字标题）用于活跃度统计
- **用户偏好 (User Preferences)**: 用户选择的模式偏好、界面布局、主题等设置
- **用量记录 (Usage Record)**: 用户的 AI 调用计量数据——调用次数、模型类型、token 消耗量、时间戳、对话摘要（前 50 字）。不包含完整对话内容。存储于后台服务器
- **管理员账户 (Admin Account)**: 通过角色标记 `admin` 区分的特权账户，由后台直接创建，与普通用户共用同一认证体系。拥有查看数据、管理邀请码、推送版本、审核 Skill 市场的权限
- **Skill 发布 (Skill Publication)**: 开发者提交的 Skill 工具，包含代码包、描述、版本号，经管理员审核后发布到市场
- **版本更新 (Version Update)**: 新版本包地址、版本号、更新日志，由管理员推送，客户端轮询或接收通知

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从启动应用到登录成功进入主界面的时间不超过 10 秒
- **SC-002**: 开发模式切换至简约模式的过渡时间不超过 1 秒
- **SC-003**: 简约模式下，用户无需阅读文档即可完成首次 AI 对话的比率达到 80%
- **SC-004**: AI 对话在 Markdown 格式回复（含代码块）下的渲染时间不超过 500 毫秒
- **SC-005**: 应用在不同平台（Windows、macOS）上功能表现一致，无平台特定崩溃
- **SC-006**: AI 功能达到 Codex 插件满血能力——包含多轮对话、代码生成与补全、代码审查、错误诊断与修复建议、文件操作、终端命令执行，用户可通过自然语言完成开发任务

## Clarifications

### Session 2026-06-28

- Q: 用户账户数据的存储和验证在哪里完成？ → A: 需要独立后台服务器，负责登录认证、计量存储、版本推送、Skill 市场分发，但不存储用户项目文件内容数据；AI 对话侧必须适配本地 Codex 插件/本地能力框架
- Q: 新用户账户的创建方式？ → A: 邀请制——管理员在后台生成邀请码，用户凭邀请码注册；不支持开放自助注册
- Q: 计量数据记录粒度？ → A: 调用统计 + 会话摘要——记录调用次数、模型类型、token 消耗、时间戳，同时存储每条对话的标题/摘要（前 50 字）用于后台活跃度查看；不记录完整对话内容
- Q: AI 对话历史（完整内容）存储位置？ → A: 直接复用 Codex 插件已有的本地对话存储机制，不自行实现；客户端调用 Codex 接口或命令桥接读写对话历史
- Q: 用户登录后可以使用哪个模式？ → A: 所有用户均可通过菜单自由切换两种模式，无权限限制；默认模式由用户偏好决定
- Q: 管理员如何登录管理后台？ → A: 同一认证体系——管理员由后台直接创建，使用相同的用户名+密码登录管理后台，通过角色标记区分权限

## Assumptions

- 基于 VSCode 开源代码（Code - OSS）构建，保留其编辑器和扩展能力，在此基础上做定制
- 目标用户分为两类：研发人员和业务人员，均可在菜单中自由切换两种模式，无角色权限限制
- 简约模式支持基础文本编辑（修改、保存），但不提供语法高亮和代码补全——高级编辑在开发模式中完成
- Codex 插件作为 AI 后端，以满血能力集成（对话、代码生成、补全、审查、错误修复、文件操作、终端命令），通过插件或本地进程与编辑器通信
- 登录方式首版采用本地账户（用户名+密码），v2 按需扩展 OAuth2 第三方登录。认证由独立后台服务器处理
- 后台服务器仅存储登录凭证、用量计量数据，不存储用户项目文件和代码内容
- 客户端 AI 对话层适配本地 Codex 插件/本地能力框架，复用其协议、命令与认证边界
- 管理后台通过 Web 页面访问，独立于编辑器客户端运行
- AI 对话完整历史数据复用 Codex 插件内置的本地存储机制，客户端不自行实现对话持久化
- 应用需要网络连接以使用 AI 功能和登录认证，基础文件浏览和编辑功能可离线使用
- 首版不支持移动端（仅 Windows/macOS 桌面端）
- 简约模式面向需要查看和基础编辑代码但不需要完整 IDE 能力的用户（如项目经理、QA、业务分析师），同时保留通过 AI 对话完成复杂操作的能力
